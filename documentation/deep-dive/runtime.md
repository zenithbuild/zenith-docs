## 👻 The Zenith Runtime

Zenith's runtime is designed to be "invisible"—a tiny, high-performance layer that manages the bridge between your state and the browser's DOM.

### Hydration: The Awakening

When a Zenith page loads, the runtime performs **Hydration**. Instead of re-rendering the entire page, it "wakes up" the static HTML delivered by the server.

1. **Placeholder Search**: It scans the DOM for `data-zen-*` attributes.
2. **Expression Linking**: It connects these placeholders to their corresponding evaluation functions.
3. **Event Binding**: It wires up high-level event handlers (like `onclick`) to the living DOM elements.

## 🔗 The Binding System

The runtime maintains a registry of bindings. Each binding consists of:
- **A Target**: The specific DOM node or attribute.
- **An Expression ID**: A unique identifier for the logic generated by the compiler.
- **Dependents**: A list of state variables that trigger an update.

### Anatomy of an Update

When state changes:
1. The property setter triggers a reactive notification.
2. The runtime identifies all bindings dependent on that property.
3. It re-evaluates the expressions and applies the results directly to the DOM target.

## ⚡ Zero-Diff Reactivity

Unlike VDOM frameworks, Zenith's runtime doesn't diff entire trees. Because it knows exactly which node depends on which state, it can skip the "diff and patch" phase entirely, jumping straight to the update.

## 🌌 Deep Dive: Proxy vs Object.defineProperty

Depending on the environment and complexity, Zenith's runtime may use different strategies for state tracking. In modern browsers, we leverage `Proxy` for its deep-tracking capabilities and performance. For legacy or constrained environments, the compiler can fall back to optimized `Object.defineProperty` chains.
